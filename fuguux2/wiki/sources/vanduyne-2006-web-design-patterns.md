# Making the Most of Web Design Patterns

Chapter 2 of *The Design of Sites*. Explains the book's design-pattern framework ([[web-design-patterns]]): what a pattern is, the six-part format used to document each pattern (name, background, problem, forces, solution, other patterns to consider), how patterns connect into a larger pattern language, and how patterns are grouped (A–M) to support different stages of the design process.

## Key points

- Patterns are traced back to architect Christopher Alexander's *A Pattern Language: Towns, Buildings, Construction* (1977), which proposed a shared language for design decisions. The authors adapt this idea to Web design, with an explicit emphasis on customer-centered design ([[customer-centered-design]]).
- A worked example, [[action-buttons]], demonstrates the full pattern format and the idea of "forces" — the competing concerns (visibility, size, download speed) a designer must balance.
- A second worked example, [[search-action-module]], shows the format applied to a more complex pattern with a background, search-selector trade-offs, and links to related patterns.
- A running example follows a designer, Sarah, using the pattern language plus customer testing (recruiting five participants, observing a checkout flow) to diagnose and redesign a broken checkout — illustrating [[process-funnel]] and [[customer-centered-design]] in practice.
- The chapter argues that core interaction patterns (navigation bars, search boxes, checkout flows) have stayed stable across major sites ([[ebay]], [[amazon]], [[google]], [[yahoo]], CNN, MSN) from 2001–2006 even as visual styling evolved, supporting the claim that the underlying patterns are durable.
